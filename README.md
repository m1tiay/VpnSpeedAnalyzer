# VPN Speed Analyzer

[English](README.en.md) | Русский

`VPN Speed Analyzer` — настольное приложение на `WPF + .NET 8` для контроля
качества VPN-подключения и выбора лучшего хоста по реальным сетевым замерам.

## Краткое руководство (для пользователя)

1. Скачайте `VpnSpeedAnalyzer-win-x64.exe` со страницы Releases: <https://github.com/m1tiay/VpnSpeedAnalyzer/releases>.
2. Скачайте `speedtest.exe` (Ookla CLI): <https://www.speedtest.net/apps/cli>.
3. Положите `speedtest.exe` рядом с `VpnSpeedAnalyzer.exe` (обязательно: без него замеры не выполняются).
4. Добавьте `speedtest.exe` в VPN-контур (split tunneling / правила маршрутизации VPN), иначе замеры могут идти мимо VPN.
5. Запустите `VpnSpeedAnalyzer.exe`, нажмите `Старт` и дождитесь первого успешного замера.

## Основная логика приложения

- Приложение делает замеры канала: `ping`, `джиттер`, `потери`, `загрузка`, `отдача`.
- По результатам считает `D.Q.S` (`0..100`) и ранжирует хосты.
- Чем выше `D.Q.S`, тем лучше хост для выбранного профиля.
- На вкладке `Мониторинг` виден текущий топ, на вкладке `Аналитика` — сводка по всем хостам.

### Как считается D.Q.S

`D.Q.S` — это итоговая оценка качества хоста по шкале `0..100`.

Логика простая:

1. Приложение берёт 5 метрик: `ping`, `джиттер`, `потери`, `загрузка`, `отдача`.
2. Для каждой метрики считает “насколько это хорошо”:
   - `ping`, `джиттер`, `потери` — чем меньше, тем лучше;
   - `загрузка`, `отдача` — чем больше, тем лучше.
3. Затем объединяет всё в один балл `D.Q.S`.

Пороговые ориентиры для оценки:

| Metric      | ideal | worst | What it measures      | Trend            |
| ----------- | ----- | ----- | --------------------- | ---------------- |
| ping (ms)   | 20    | 150   | latency               | lower is better  |
| jitter (ms) | 2     | 40    | latency stability     | lower is better  |
| loss (%)    | 0     | 5     | packet loss           | lower is better  |
| download    | 400   | 20    | download speed (Mbps) | higher is better |
| upload      | 150   | 10    | upload speed (Mbps)   | higher is better |

Почему один и тот же хост может иметь разный итог в разных режимах:

- в профиле меняется важность метрик (их вклад в общий балл);
- сами замеры не меняются, меняется только то, на что делать акцент.

Как читать результат:

- `90+` — очень хороший хост для выбранного сценария;
- `70-90` — рабочий стабильный вариант;
- ниже `70` — стоит сравнить с другими хостами в таблице.

## Логика профилей

Профиль — это ответ на вопрос: **“что для вас важнее прямо сейчас?”**  
При смене профиля приложение не “подкручивает” результаты, а по-другому расставляет приоритеты и сразу пересчитывает рейтинг хостов.

Принцип очень простой:

- если нужен баланс — выше будет самый ровный вариант без явных провалов;
- если важна отзывчивость — выше будет хост с лучшим откликом;
- если важна скорость — выше будет хост с лучшими скоростями.

Как это выглядит в интерфейсе:

- `Универсальный` — сбалансированный режим.
- `Игры` — приоритет минимальной задержки и потерь.
- `Стрим` — приоритет стабильности и скоростей.

Что реально меняется при переключении профиля:

- меняется “вес” метрик в итоговой оценке `D.Q.S`;
- один и тот же хост может быть №1 в одном профиле и опуститься ниже в другом;
- сами сырые замеры (`ping`, `джиттер`, `потери`, `загрузка`, `отдача`) не подменяются — меняется только то, **что считать важнее**.

Простой пример:

- в `Универсальный` чаще побеждает хост без явных слабых мест по всем метрикам;
- в `Игры` хост с низким пингом обычно поднимется выше, даже если у него чуть ниже скорость;
- в `Стрим` хост с более высокими скоростями может стать лидером, если задержка остаётся в разумных пределах.

Практически:

- переключайте профиль под текущую задачу;
- смотрите, как меняется лидер;
- выбирайте хост не “навсегда”, а под конкретный сценарий (игры, стрим, обычная работа).

## Логика отслеживания смены хоста VPN

Приложение отслеживает transport fingerprint VPN-процесса (PID-based):

- изменение оценивается через `delta` (разница endpoint-ов);
- применяется `debounce` (~2.5 сек), чтобы отсеять краткий шум;
- события делятся на:
  - `PRIMARY` — крупная реальная смена (обычно ручная или auto-server),
  - `TAIL` — внутренние хвостовые перестройки.

**Автозамер запускается только для `PRIMARY`.**  
Это снижает ложные срабатывания, но оставляет реакцию на реальную смену хоста.

## Основные функции в одном месте

- Мониторинг качества VPN в реальном времени.
- Рейтинг и аналитика хостов по `D.Q.S`.
- Профили под разные сценарии (`Универсальный`, `Игры`, `Стрим`).
- Автодетект смены хоста VPN с защитой от ложных срабатываний.
- Экспорт CSV (замеры и топ хостов).

## Сборка релиза (Single EXE)

В GitHub Actions выполняется публикация standalone EXE для Windows (`win-x64`) с загрузкой артефакта.

Локальная команда публикации:

```
dotnet publish VpnSpeedAnalyzer.csproj -c Release -r win-x64 --self-contained true -o out
```

## Технологии

- **Платформа:** `WPF`, `.NET 8`
- **Графики:** `ScottPlot`
- **Архитектура:** MVVM-подход (`ViewModels`, `Models`, `Logic`, `Services`)
- **CI/CD:** GitHub Actions
- **Дистрибуция:** standalone single EXE

## Логи и приватность

- Логи: `%LOCALAPPDATA%\VpnSpeedAnalyzer\logs` (кратко: файл на запуск, автоочистка).
- Замеры делает локально установленная `speedtest.exe` — внешние сервисы используются только для определения IP/страны/ASN.

## Лицензия сторонних компонентов

- `speedtest.exe` (Ookla Speedtest CLI) является сторонним компонентом Ookla.
- Пользователь скачивает его отдельно с официального сайта: <https://www.speedtest.net/apps/cli>.
- Использование `speedtest.exe` регулируется условиями и лицензией Ookla; приложение не изменяет и не переупаковывает этот бинарник.
